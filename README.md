# codegen-standards

## О репозитории

`codegen-standards` — это набор skills и правил, которые задают единый инженерный стандарт для AI-генерации.

## Bootstrap Mode

Если проект в ранней фазе, используйте `*-LINTING.md` и `*-ARCHITECTURE.md` как инструкцию по запуску, а не как полный strict-набор с первого дня.

Рекомендуемый порядок:
1. Установить зависимости из `*-LINTING.md` и настроить базовый ESLint/Prettier/Husky.
2. Создать целевую структуру папок из `*-ARCHITECTURE.md` (например `features/*/{ui,model,api,lib}`), даже если часть папок пока пустая.
3. Оставить минимальный набор обязательных правил:
- TypeScript safety (`no any`, promises, null checks)
- Hooks safety (`rules-of-hooks`, `exhaustive-deps`)
- базовый import/order
- naming consistency (например `PascalCase`, `useX`, `*.api.ts`)
4. Временно отключить или ослабить часть правил, которые тормозят старт (сложные stylistic/strict ограничения).
5. По мере роста проекта постепенно включать остальные CORE/RECOMMENDED правила.

Важно:
- Даже в bootstrap-режиме финальный код должен быть предсказуемым и единообразным.
- Ослабление правил должно быть явным, документированным и обратимым.

Production-grade стандарты для AI-агентов, генерирующих код.

Репозиторий нужен, чтобы AI:
- писал код предсказуемо и последовательно;
- соблюдал архитектурные ограничения проекта;
- выдавал результат, пригодный для поддержки и production.

## Зачем это нужно

Без стандартов AI часто генерирует разнородные решения: разный стиль, случайные архитектурные ходы, неполные реализации.

С этим репозиторием команда получает:
- единое качество кода между людьми и агентами;
- меньше правок после генерации;
- прозрачные правила, которые можно обсуждать, версионировать и улучшать.

## Как использовать

Подключайте skills явно перед генерацией кода и указывайте, что правила обязательны к соблюдению.

Базовое правило:
- `*-CORE.md` для выбранного стека должен участвовать всегда. Это основной файл, который задаёт инварианты генерации кода.
- `*-ARCHITECTURE.md` подключается вместе с `CORE`, когда важны границы модулей, слоёв и структура проекта.
- `*-LINTING.md` подключается вместе с `CORE`, когда важны правила оформления, линтинга и проверяемая консистентность.

Рекомендуемый процесс:
1. Определите контекст задачи (стек, слой, тип фичи).
2. Сначала подключите `CORE` для нужного стека.
3. Затем добавьте `ARCHITECTURE` и `LINTING`, если задача требует архитектурных ограничений и строгих правил оформления.
4. Зафиксируйте приоритет: `CORE` -> архитектура -> реализация -> стиль/линтинг.
5. Проверьте результат на соответствие правилам перед merge.

Рекомендуемые наборы по стеку:
- `React`: `REACT-CORE.md` + при необходимости `REACT-ARCHITECTURE.md` + `REACT-LINTING.md`
- `Next.js`: `NEXT-CORE.md` + при необходимости `NEXT-ARCHITECTURE.md` + `NEXT-LINTING.md`
- `Vue`: `VUE-CORE.md` + при необходимости `VUE-ARCHITECTURE.md` + `VUE-LINTING.md`
- `Nest`: `NEST-CORE.md` + при необходимости `NEST-ARCHITECTURE.md` + `NEST-LINTING.md`
- `Express`: `EXPRESS-CORE.md` + при необходимости `EXPRESS-ARCHITECTURE.md` + `EXPRESS-LINTING.md`

Важно: skills не заменяют ревью. Они уменьшают вариативность и повышают качество первого результата.

## Принципы

- Architecture-first: сначала границы и ответственность модулей, потом код.
- Explicit rules: требования формулируются как проверяемые правила.
- Deterministic output: одинаковый контекст должен давать близкий результат.
- Senior mindset: решения с учетом поддержки, масштабирования и DX.

## Кому подходит

- Командам, где AI участвует в ежедневной разработке.
- Проектам с высокими требованиями к качеству и консистентности.
- Техлидам, которым нужен управляемый стиль AI-генерации.
